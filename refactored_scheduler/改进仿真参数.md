# 改进仿真参数说明

## 1. 修改目标

本次只先确认参数与生成逻辑理解，暂不改代码。

目标是在现有 `refactored_scheduler` 的流量生成模块和主流程中，把 TS 流从原来的“两类流”扩展为四类流：

- 等时流 1：`Iso_Type1`
- 等时流 2：`Iso_Type2`
- 循环流 1：`Cli_Type1`
- 循环流 2：`Cli_Type2`

四类 TS 流各占总流量数量的 `1/8`，BE 流占剩余 `1/2`。因此总数量比例为：

| 流类型 | 数量占比 |
|---|---:|
| `Iso_Type1` | `1/8` |
| `Iso_Type2` | `1/8` |
| `Cli_Type1` | `1/8` |
| `Cli_Type2` | `1/8` |
| `BE_Type` | `1/2` |

合计为 `1/8 * 4 + 1/2 = 1`。

## 2. “总数”的理解

建议保留现有负载参数 `traffic_load = 0.1` 和链路速率 `R = 49.7664e9`，仍用原来的数量基准公式推导总包/流数量：

```python
total_expr = (R * traffic_load) / 2.67e6
total_flows = math.ceil(total_expr)
```

然后按比例拆分：

```python
num_Iso_Type1 = ceil(total_flows * 1 / 8)
num_Iso_Type2 = ceil(total_flows * 1 / 8)
num_Cli_Type1 = ceil(total_flows * 1 / 8)
num_Cli_Type2 = ceil(total_flows * 1 / 8)
num_be_packets = total_flows - (
    num_Iso_Type1 + num_Iso_Type2 + num_Cli_Type1 + num_Cli_Type2
)
```

这样 BE 数量自动吃掉整数取整后的剩余部分，确保五类数量加起来严格等于 `total_flows`。

如果希望 BE 也强制 `ceil(total_flows * 1 / 2)`，则总数可能因为取整超过 `total_flows`。我建议采用“TS 四类先按 1/8 取整，BE 取剩余”的方式。

## 3. 流类型参数保持现有配置

现有四类 TS 流参数继续沿用：

| 流类型 | cycle(us) | delay(us) | jitter(us) | size(byte) |
|---|---:|---:|---:|---:|
| `Iso_Type1` | 600 | 500 | 1 | 200 |
| `Iso_Type2` | 1200 | 1000 | 1 | 400 |
| `Cli_Type1` | 2400 | 1000 | 1000 | 800 |
| `Cli_Type2` | 4800 | 2000 | 1500 | 1600 |

超周期建议由四类周期共同计算：

```python
supercycle_size = lcm(600, 1200, 2400, 4800)
```

结果仍为 `4800 us`。

## 4. 固定随机种子

需要固定所有随机生成，保证每次运行结果可复现。

建议在 `config.py` 中增加统一随机种子：

```python
TS_RANDOM_SEED = 20260709
BE_RANDOM_SEED = 123
```

TS 流的 `start_time` 生成不再直接使用全局 `np.random.randint`，而是使用固定随机生成器：

```python
rng = np.random.default_rng(TS_RANDOM_SEED)
start_time = int(rng.integers(0, flow_params[flow_type]["cycle"]))
```

BE 流生成函数也改为固定种子输入，默认使用 `BE_RANDOM_SEED`。

## 5. 流量生成逻辑

主流程中应按四类 TS 流分别生成，而不是只生成 `Iso_Type1` 和 `Cli_Type1`。

建议抽出统一函数，例如：

```python
generate_ts_flows(flow_counts, flow_params, num_onu, seed)
```

输入示例：

```python
flow_counts = {
    "Iso_Type1": num_Iso_Type1,
    "Iso_Type2": num_Iso_Type2,
    "Cli_Type1": num_Cli_Type1,
    "Cli_Type2": num_Cli_Type2,
}
```

生成出的每条 TS 流仍包含现有字段：

- `flow_id`
- `onu_id`
- `cycle`
- `size`
- `delay`
- `jitter`
- `start_time`
- `flow_type`
- `num_flows`
- `flow_ids`

其中 `onu_id` 仍可沿用 `i % NUM_ONU` 均匀分配。

## 6. 同周期合并逻辑

现有同周期合并逻辑可以保留：

- `Iso_Type1`、`Iso_Type2` 都归一化为 `isochronous`，不合并，逐条透传。
- `Cli_Type1`、`Cli_Type2` 都归一化为 `cyclic`，按 `(onu_id, cycle)` 分组合并。
- 合并后仍保留 `_orig_flow_type`，最终调度和统计表中能区分 `Cli_Type1` 与 `Cli_Type2`。

注意：循环流 1 和循环流 2 的周期不同，因此即使同 ONU，也会分到不同 cycle 组，不会互相合并。

## 7. 调度与排序

ILP 调度函数 `optimize_schedule` 已经按每条流的 `cycle / delay / jitter / size` 展开调度单元，因此支持四类 TS 流，不需要改模型结构。

需要调整的是输入列表：

- 原来只包含 `Iso_Type1` 和 `Cli_Type1`
- 改为包含 `Iso_Type1`、`Iso_Type2`、`Cli_Type1`、`Cli_Type2`

调度结果排序建议在输出最终表前按以下字段排序：

```python
["调度开始时间_num", "ONU_ID", "流ID"]
```

失败项没有数值型开始时间时排在后面。

## 8. 统计口径

统计需要适配新的四类 TS 参数：

1. 总体成功率可以继续按 `Iso`、`Cli` 大类统计。
2. 也建议新增细分类统计：`Iso_Type1`、`Iso_Type2`、`Cli_Type1`、`Cli_Type2`、`BE_Type`。
3. 聚合流统计仍按 `num_flows` 加权，避免同周期合并后低估原始循环流数量。
4. DJSR、平均延迟、成功数量、失败数量都应使用新的 `flow_counts` 作为基准。

建议 Excel 输出中新增或更新统计表：

- `延迟统计_原始流加权`
- `成功率_RUE_DJSR`
- `细分类成功率统计`
- `流量生成参数`

## 9. BE 流

BE 包数量使用剩余 `1/2`。

BE 仍通过现有泊松到达过程生成：

```python
generate_be_traffic(be_lambda, num_be_packets, seed=BE_RANDOM_SEED)
```

BE 不参与 TS 同周期合并和 ILP，而是在 TS 二次调度完成后，继续使用现有 `schedule_be_flows_based_on_ts` 进行空闲区间补调度。
