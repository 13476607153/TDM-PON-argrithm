# 重构调度器代码说明

本文是 `refactored_scheduler` 唯一且完整的技术说明，以当前代码实现为准，覆盖模块职责、配置参数、统一流量生成、1 μs 资源模型、核心调度算法、统计口径、输出、运行方法和测试。

当前系统的核心目标是：生成四类 TS 周期流和一类 BE 流，先对 TS 流进行同周期聚合，再通过 ILP 完成整流接纳和 DBA 周期粗分配，随后在统一的 1 μs 全局链路日历中完成精确放置，最后利用剩余时隙调度 BE 包。

当前目录结构：

```text
refactored_scheduler/
├─ __init__.py                 包声明
├─ __main__.py                 标准模块启动入口
├─ run_refactored_scheduler.py 兼容启动入口
├─ config.py                   参数和流量数量推导
├─ traffic.py                  TS/BE 输入生成与聚合
├─ time_model.py               离散时间和时间窗
├─ ts_optimizer.py             TS 第一阶段 ILP
├─ slot_calendar.py            共享日历、TS 精调度、BE 补调度
├─ statistics.py               统计指标
├─ plotting.py                 可视化
├─ main.py                     主流程编排
├─ tests/                      回归测试包
└─ 代码说明.md                 唯一技术文档
```

# 第一部分：各模块功能

## 1. 总体处理流程

程序执行链路如下：

```text
读取全局参数
  ↓
生成四类 TS 周期流
  ↓
Iso 流透传、Cli 流按 ONU 和周期聚合
  ↓
构造超周期内的 TS 实例及离散时间窗
  ↓
ILP：按整条流接纳，并为每个实例分配 DBA 周期
  ↓
全局 1 μs 日历：按整条流精确放置 payload 和 guard
  ↓
BE 包在同一日历剩余空间中补调度
  ↓
生成统计表、Excel 文件和延迟分析图
```

所有 ONU、所有 TS 类型以及 BE 共用一个链路日历，因此任意时刻只能由一个调度单元占用。每个成功调度单元至少占用 1 μs payload，并在其后占用 1 μs guard。

## 2. `run_refactored_scheduler.py`：外部启动入口

该文件是适用于 PyCharm 或直接命令行运行的薄入口。

- 导入 `refactored_scheduler.main.main`；
- `run_refactored_scheduler()` 调用主函数并返回进程退出码；
- 作为脚本运行时，通过 `SystemExit` 把退出码返回操作系统。

该模块不包含调度逻辑。运行时应保证项目父目录位于 Python 模块搜索路径中。

## 3. `config.py`：全局配置与派生参数

该模块集中定义：

- 链路速率、ONU 数量和物理时延；
- 1 μs 离散资源模型参数；
- TS 四种细分类型的周期、时延、抖动和包长；
- payload-only 负载及四类 TS 流、BE 包的数量推导；
- 随机种子；
- 超周期和每类周期流在超周期内的实例数；
- 同周期聚合的最大容量。

其他模块通过导入配置值保持统一时间和资源语义。`supercycle_size` 由四类 TS 周期的最小公倍数得到，当前为 4800 μs。

## 4. `traffic.py`：流量生成与同周期聚合

### 4.1 `generate_ts_flows`

按照 `flow_counts` 依次生成 `Iso_Type1`、`Iso_Type2`、`Cli_Type1` 和 `Cli_Type2`。

每条 TS 原始流包括：

- `flow_id`：全局递增流 ID；
- `onu_id`：按 `i % num_onu` 均匀分配；
- `cycle`、`size`、`delay`、`jitter`：来自类型配置；
- `start_time`：在 `[0, cycle)` 内随机生成；
- `flow_type`：细分流类型；
- `num_flows=1`、`flow_ids=[flow_id]`：为后续聚合统计保留映射。

函数使用局部 `numpy.random.default_rng`，不会修改 NumPy 全局随机状态，相同种子可重复得到相同流量。

### 4.2 `generate_be_traffic`

BE 包数先由 payload-only 负载公式确定。给定包数后，函数使用固定随机种子在 `[0,4800)` 内生成均匀样本并排序；这等价于在给定事件总数条件下的齐次泊松过程到达时刻。包长在 64～1500 byte 之间随机生成，再按有效链路速率换算为物理传输时间，ONU ID 独立随机生成。

参数 `lambda_val` 仅为兼容旧调用签名而保留，当前不参与计算。函数返回到达时间数组、物理传输时长数组和 ONU ID 数组；相同种子、相同包数必然得到完全相同的结果。

### 4.3 `same_cycle_merge`

聚合规则为：

- Iso 流不合并，逐条透传；
- Cli 流按 `(onu_id, cycle)` 分组；
- 同组流累计包长，超过 `MAX_MERGE_SIZE` 时拆成新的聚合桶；
- 聚合流的 `delay` 和 `jitter` 取组内最严格值；
- `start_time` 取组内最小值；
- 保存 `num_flows`、`flow_ids` 和原始 `flows`。

主流程在调用前把细分类型临时归一化为 `isochronous` 或 `cyclic`，同时用 `_orig_flow_type` 保存原类型。由于 `Cli_Type1` 和 `Cli_Type2` 周期不同，它们不会被合并到同一个周期组。

## 5. `time_model.py`：统一离散时间模型

该模块是 ILP、全局日历和资源统计的时间语义基础。

### 5.1 `transmission_time_us`

计算物理发送时间：

```text
raw_tx_time_us = size_bytes × 8 / link_rate_bps × 10^6
```

### 5.2 `resource_slots`

将物理发送时间转换为离散资源占用：

```text
payload_slots = max(MIN_PAYLOAD_SLOTS, ceil(raw_tx_time_us / SLOT_SIZE_US))
occupied_slots = payload_slots + guard_slots
```

当前包长和 50 Gbit/s 链路下，通常得到 1 个 payload slot 加 1 个 guard slot，总占用 2 μs。

### 5.3 `build_time_window`

根据到达时间、业务时延上限及系统时延预算生成：

- `release_slot`：最早允许开始的整数 slot；
- `service_deadline_slot`：payload 必须完成的截止 slot；
- `latest_start_slot`：满足 deadline 的最晚开始 slot。

当前实现沿用旧模型预算：

```text
release_time = arrival + synchronization_error
deadline = arrival + delay_limit
           - 2 × synchronization_error
           - 2 × processing_delay
           - propagation_delay
latest_start = deadline_slot - payload_slots
```

业务 deadline 只约束 payload 完成；guard 必须占用资源，但不要求在业务 deadline 前完成。

### 5.4 `eligible_dba_periods` 与 `effective_window`

前者枚举一个实例可以进入的 DBA 周期。后者把实例自身时间窗与指定 DBA 周期求交，并保证 payload 与 guard 都不跨 DBA 边界。

## 6. `ts_optimizer.py`：TS 整流接纳与 DBA 粗分配

### 6.1 `_build_units`

把每条周期流展开为超周期内的多个实例。实例数量为：

```text
supercycle_size / cycle
```

每个实例记录流 ID、ONU、周期序号、到达时间、细分类型、时延与抖动约束、聚合映射、资源 slot 数、时间窗及候选 DBA 周期。

### 6.2 `optimize_schedule`

使用 DOcplex 构造混合整数线性规划：

- `accept[f]`：整条 TS 流是否接纳；
- `z[u,d]`：实例 `u` 是否分配给 DBA 周期 `d`；
- `dmax`：不同 DBA 周期负载的最大差异。

整流接纳约束要求同一流的每个实例都随 `accept[f]` 一起接纳：

```text
Σ z[u,d] = accept[f]
```

若一条流中任一实例没有候选 DBA 周期，则该流不能接纳。

每个 DBA 周期的资源约束使用 `occupied_slots`，即 payload 和 guard 的总占用。代码还加入关键截止点前缀容量约束，减少 ILP 总容量可行但精确日历无法放置的情况。

目标函数首先最大化按 `num_flows` 加权的原始 TS 流接纳量，其次通过惩罚 `dmax` 平衡 DBA 周期负载。ILP 只决定接纳和 DBA 归属，不决定最终开始时间。

## 7. `slot_calendar.py`：全局 1 μs 精确调度

### 7.1 `LinkCalendar`

全局日历使用长度为 `supercycle_size` 的列表：

```text
None                 空闲
("PAYLOAD", token)  payload 占用
("GUARD", token)    guard 占用
```

主要操作包括：

- `is_free(start, end)`：检查半开区间 `[start,end)` 是否空闲；
- `reserve(unit, start)`：连续写入 payload 和 guard；
- `release_flow(flow_id)`：回滚一条流的全部实例；
- `occupied_count()`：统计已占用 slot 数。

### 7.2 `schedule_ts_units`

该函数执行第二阶段精确调度，并保持整流事务语义。

流排序依据依次为：

1. 抖动限制升序；
2. 时间窗宽度升序；
3. 最早截止时间升序；
4. 总占用 slot 降序；
5. 流 ID 升序。

对每条流，算法枚举首个实例的候选开始位置，由此得到统一相位；其余实例优先搜索最接近目标相位的空闲起点。只有全部实例成功放置且实际抖动不超过限制时才提交。

若任一实例冲突或抖动超限，调用 `release_flow` 回滚该流已占用的全部 slot。失败原因当前区分：

- `CALENDAR_CAPACITY_CONFLICT`：日历容量或时间窗冲突；
- `JITTER_VIOLATION`：实例均找到位置，但整流抖动超限。

### 7.3 `refine_ts_schedule`

该函数把 ILP 输出转为日历调度输入，调用 `schedule_ts_units` 后补充结果表需要的 `duration`、`slot`、`slot_size`、`slot_start`、`slot_end` 和 `status` 字段。原 `secondary_scheduler.py` 只是这一过程的兼容壳，现已合并并删除。

### 7.4 `schedule_be_packets`

BE 与 TS 复用同一个 `LinkCalendar`。每个 BE 包按到达时间稳定排序，从向上取整后的到达 slot 开始进行首次适配搜索。原 `be_scheduler.py` 已合并并删除。

算法规则：

- payload 至少占 1 个 slot；
- 每个包后增加 `GUARD_SLOTS`；
- payload 和 guard 均不得跨 DBA 周期边界；
- 不移动已经成功放置的 TS；
- 搜索至超周期末尾仍无空间则返回失败。

函数也兼容旧调用：如果没有传入已占用日历，可把旧格式 TS 时间窗转换为日历占用。

## 8. `statistics.py`：统计模块

统计模块通过 `聚合原始流数量` 或内部 `num_flows` 实现原始流加权，避免 Cli 聚合后把多条原始流误算成一条。

主要功能：

- `calculate_jitter_statistics`：按 TS 流计算实际抖动及最差流；
- `build_weighted_delay_stats`：生成各细分类型和总体的加权延迟表；
- `build_flow_type_success_stats`：输出五种类型的生成数、成功数、失败数、成功率和延迟；
- `compute_statistics`：生成 Iso/Cli 大类成功率、DJSR、RUE 和资源利用指标。

流量指标与资源指标采用不同口径：

- 成功数、失败数、平均延迟等流量指标按 `num_flows` 加权；
- payload、guard 和物理传输时间等链路资源只按实际聚合调度单元计算，不能再次乘 `num_flows`。

## 9. `plotting.py`：图表输出

该模块生成 `延迟分布分析.png`，主要包含：

- Iso、Cli、BE 的延迟小提琴图；
- TS 成功实例在各时间桶中的数量；
- TS 聚合实例成功/失败比例；
- 控制台输出 BE 成功率、延迟和 TS 抖动摘要。

## 10. `main.py`：主流程编排

`main()` 负责串联所有模块：

1. 生成四类 TS 流并保存参数快照；
2. 归一化类型并执行同周期聚合；
3. 根据 `USE_SHAPING` 选择聚合流或原始流作为调度输入；
4. 调用 ILP 完成整流接纳和 DBA 粗分配；
5. 调用全局日历完成 TS 精确放置；
6. 按固定种子生成超周期内的 BE 流并在同一日历中补调度；
7. 统一生成中文结果字段；
8. 计算排队延迟和完成时延并排序；
9. 输出 Excel、图像及控制台统计。

当前 Excel 文件名为 `完整调度结果_含聚合映射_原始流加权统计.xlsx`。若文件正在 Excel/WPS 中打开，Windows 会拒绝覆盖并产生 `PermissionError`。

## 11. `tests/`：核心不变量与流量一致性

当前测试覆盖：

- 小包至少获得 1 个 payload slot 和 1 个 guard slot；
- TS 整流精确放置后 payload/guard 数量正确且抖动满足约束；
- BE 使用与 TS 相同的日历，并避开 TS 已占用空间。

`tests/test_traffic.py` 还覆盖：

- 四类 TS 包长均为 6220 byte，物理发送时间不超过 1 μs；
- `rho=0.1～0.9` 时四类 TS 流数相同；
- TS 包实例与 BE 包之和严格等于 `round(rho×4800)`；
- 固定种子的 TS 和 BE 输入可复现；
- `lambda_val` 不改变给定包数条件下的 BE 样本。

| 参数 | 当前值 | 单位 | 说明 |
|---|---:|---|---|

## 15. 离散时间和 DBA 参数

| 参数 | 当前值 | 说明 |
|---|---:|---|
| `SLOT_SIZE_US` | 1 μs | 全局最小资源粒度 |
| `MIN_PAYLOAD_SLOTS` | 1 | 每个单元至少占用的 payload slot 数 |

| `DBA_PERIOD_US` | 300 μs | DBA 周期长度 |

内部所有离散区间使用半开区间 `[start,end)`。例如从 slot 10 开始、payload 为 1、guard 为 1，则 payload 占 `[10,11)`，guard 占 `[11,12)`。

## 16. 流量负载与数量参数

当前配置：

```text
traffic_load = 0.9
target_payload_packets = round(traffic_load × 4800)
```

`traffic_load` 只计算 payload slot，guard 由调度器产生，不从流量预算中预扣。目标 payload 包实例约一半为 TS、一半为 BE；四类 TS 的流数量相等：

```text
packets_per_equal_ts_group = 8 + 4 + 2 + 1 = 15
count_per_ts_type = floor((target_payload_packets / 2) / 15)
ts_payload_packets = count_per_ts_type × 15
num_be_packets = target_payload_packets - ts_payload_packets
```

在实验负载 `rho=0.1～0.9`（步长 0.1）下可精确二分。例如 `rho=0.9` 时，每类 TS 144 条流、TS 2160 个包实例、BE 2160 个包，总 payload 为 4320 个 1 μs slot。

| 类型 | 数量规则 | 说明 |
|---|---:|---:|
| `Iso_Type1` | `count_per_ts_type` | 与另外三类 TS 流数相同 |
| `Iso_Type2` | `count_per_ts_type` | 与另外三类 TS 流数相同 |
| `Cli_Type1` | `count_per_ts_type` | 与另外三类 TS 流数相同 |
| `Cli_Type2` | `count_per_ts_type` | 与另外三类 TS 流数相同 |
| `BE_Type` | 目标 payload 包数减 TS 包实例数 | 按包计数 |

## 19. 同周期聚合算法

聚合的目的不是删除业务流，而是减少参与优化的调度实体数量。聚合后的 `num_flows` 表示承载的原始流数，`flow_ids` 保存来源。

聚合容量上限为：

```text
MAX_MERGE_SIZE = time_slot_size × LINK_RATE / 8 / 10^6
```

当前约为 6250 byte。若同一 ONU、同一周期的 Cli 流累计包长超过该值，则拆为多个聚合流。

由于当前每个 TS 包为 6220 byte，两个 TS 包合并后会超过 6250 byte，因此当前参数下每个聚合桶实际只能容纳一条原始 TS 流；聚合框架仍保留，以便后续包长或容量参数变化时使用。

聚合后的资源只占一次 payload 和一次 guard；但成功率和流数量按 `num_flows` 恢复到原始流口径。

## 20. TS 第一阶段 ILP 算法

### 20.1 决策变量

```text
accept[f] ∈ {0,1}
z[u,d] ∈ {0,1}
dmax ≥ 0
```

### 20.2 主要约束

整流接纳：

```text
对流 f 的每个实例 u：Σd z[u,d] = accept[f]
```

DBA 总容量：

```text
Σu occupied_slots[u] × z[u,d] ≤ DBA_PERIOD_US
```

截止点前缀容量：

```text
必须在截止点 q 前完成的实例总占用 ≤ q - DBA周期起点
```

前缀约束是日历可行性的必要条件，用于降低第二阶段因时间窗过于集中导致的失败。

### 20.3 目标函数

```text
maximize
(supercycle_size + 1) × Σf num_flows[f] × accept[f]
- dmax / total_weight
```

大权重保证优先接纳更多原始 TS 流，负载均衡只作为次级目标。

## 21. TS 第二阶段全局日历算法

ILP 输出的实例已经确定 DBA 周期。精确调度再计算有效窗口：

```text
low  = max(release_slot, DBA起点)
high = min(latest_start_slot, DBA终点 - occupied_slots)
```

对一条流的首个实例枚举 `first_start`，并计算：

```text
phase = first_start - first_arrival
target[n] = round(arrival[n] + phase)
```

每个实例的候选起点按“离 target 的距离、起点值”排序。候选区间必须连续空闲 `occupied_slots` 个 slot。

实际流抖动定义为：

```text
max(start[n] - arrival[n]) - min(start[n] - arrival[n])
```

只有所有实例放置成功且实际抖动不超过配置上限，整条流才成功；否则回滚全部实例。

## 22. BE 调度算法

BE 不进入 ILP，采用按到达时间的首次适配：

1. `release_slot = ceil(arrival_time)`；
2. 计算至少 1 个 payload slot 和 1 个 guard slot；
3. 从 release 开始逐 slot 搜索；
4. 若候选占用跨 DBA 边界，直接跳到下一个 DBA 周期起点；
5. 找到连续空闲区间后写入全局日历；
6. 到超周期末尾仍未找到则失败。

该算法保证 BE 不与 TS 冲突，也不与其他 BE 冲突。

## 23. 延迟、抖动与成功口径

结果中保留两个延迟概念：

```text
排队延迟 = 调度开始时间 - 到达时间
完成时延 = payload结束时间 - 到达时间
```

当前 `延迟(us)` 列和多数已有统计使用排队延迟；`完成时延(us)` 单独输出。设计方案要求业务 delay 最终以完成时延判定，阅读实验结果时不应混淆两者。

TS 采用整流成功口径：超周期内全部实例均成功且满足抖动，整条流才成功。BE 采用逐包成功口径。

## 24. 资源统计口径

每个成功调度单元可记录：

- `原始传输时长(us)`：物理线路发送时间；
- `包发送占用(us)`：离散 payload slot 占用；
- `隔离带占用(us)`：guard slot 占用；
- `资源占用(us)`：payload 与 guard 总和。

推荐指标含义：

```text
Payload 利用率 = Σpayload_slots / supercycle_slots
链路占用率 = Σoccupied_slots / supercycle_slots
Guard 开销率 = Σguard_slots / Σoccupied_slots
物理有效载荷率 = Σraw_tx_time_us / supercycle_size
```

聚合流的资源占用不能按 `num_flows` 重复加权，否则会高估链路占用。

## 25. 失败原因

当前代码直接产生的主要失败原因包括：

| 原因 | 含义 |
|---|---|
| `CALENDAR_CAPACITY_CONFLICT` | 有效时间窗内没有足够连续空闲 slot |
| `JITTER_VIOLATION` | 找到实例位置但整条流实际抖动超限 |
| `无可用时间窗` | BE 搜索到超周期结束仍无法放置 |

方案文档还定义了 `NO_ELIGIBLE_DBA`、`ILP_REJECTED`、`DEADLINE_VIOLATION`、`LOCAL_REPAIR_EXHAUSTED` 等更细原因；当前 ILP 被拒绝的流没有加入最终失败表，因此这些原因尚未全部显式输出。

## 28. 当前实现与设计方案的注意事项

1. `main.py` 调用 ILP 和 BE 时仍把 DBA 周期写为 `300`，其数值与 `DBA_PERIOD_US` 一致；后续修改周期时需同步调整。
2. `ALLOW_CROSS_DBA` 当前未作为条件分支使用，算法始终禁止调度单元跨 DBA。
3. `be_lambda` 是旧接口兼容参数，当前不会改变 BE 到达样本；负载完全由目标包数控制。
4. `compute_statistics` 的 DJSR 大类计算通过首个匹配类型取得限制；四种细分类型约束不同，严格统计应按具体 `flow_type` 分别判断后汇总。
5. ILP 拒绝的流当前没有全部进入最终失败表，更完整的结果可补充 `ILP_REJECTED` 等记录。
6. 当前多数延迟统计沿用排队延迟，而业务 deadline 按 payload 完成时间构造；实验报告必须明确采用的延迟口径。

以上注意事项不改变当前主流程的基本结构，但会影响结果追溯和严格的指标解释。
